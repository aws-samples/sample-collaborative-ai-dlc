# V2 Graph Context

V2 keeps upstream AI-DLC workflow files and agent-authored markdown artifacts as the canonical source of truth. The runtime derives a fine-grained graph projection from those approved artifacts so agents can query sections, typed items, and citations without loading whole markdown documents. Agents never author graph topology directly — deterministic parsers own every derived node and edge, and the whole projection is re-derivable from the artifact markdown at any time.

## Lifecycle

1. **Authoring (structure contracts).** For every output artifact type registered in the extraction registry (`lambda/shared/artifact-extractors.js` — `requirements`, `stories`, `personas`, `components`, `decisions`, `unit-of-work-story-map`, `unit-of-work-dependency`), the stage prompt injects a **structure contract** (`lambda/shared/artifact-structure-contract.js`): the expected fenced YAML block, its field specs, and a canonical example. The contracts are GENERATED from the same registry the parser reads, and a round-trip test feeds every rendered example back through the extractor — instructions and parser cannot drift. Generic rules ride along: ≥2 `##` section headings and `[[artifact-slug]]` citations for consumed inputs. The `unit-of-work-dependency` artifact carries two blocks: `units:` (the scheduling DAG — REQUIRED, enforced by a blocking sensor) and `contracts:` (inter-unit interface contracts).

2. **Derivation.** After every successful artifact-producing stage — once-per-workflow stages (orchestrator hook) AND unit-lane stages (section hook, unit-attributed) — the runtime runs `derive-artifacts`: markdown headings become `Section` nodes, registered fenced blocks become typed item nodes (`Story`, `Requirement`, `Persona`, `Component`, `Decision`, `StoryMapEntry`, `Contract`) with `HAS_ITEM`/`DERIVED_FROM` edges, and `[[slug]]` references become `CITES` edges between artifacts. Idempotent: rows upsert by deterministic ids; removed sections/items are marked superseded (never deleted); derivations of rewind-superseded artifacts are swept. A derive failure is an event, never a stage failure.

3. **Item↔item traceability edges.** After the mirrors, an intent-wide sweep (`resolveDerivedItemEdges`) materializes the slug references the structure contracts already require as typed edges: `Story --COVERS--> Requirement` (`covers`), `Story --FOR_PERSONA--> Persona` (`persona`), `Story/Component --DEPENDS_ON--> Story/Component` (`depends_on`), `StoryMapEntry --IMPLEMENTS--> Story|UnitOfWork` (`stories`/`unit`), and `UnitOfWork --EXPOSES/CONSUMES_CONTRACT--> Contract` (`provider`/`consumers`). Drop-then-recreate per current source vertex, so a re-derive that removed a reference also removes its edge; dangling slugs are skipped silently (the coverage sensor reports them as `unknownReferences`). The sweep also runs after `promote-units`, because `StoryMapEntry`/`Contract` items derive before the `UnitOfWork` vertices exist — unit wiring resolves on that later pass.

4. **Reading.** Reads are compact-first (the MCP annex teaches the ladder): `get_intent_graph` → `get_artifact_toc` / `get_artifact(mode: summary|toc)` → `get_section` / `get_items` → `search_graph` → full `get_artifact` last. `get_coverage` answers the join questions in one call: must-have requirements uncovered by stories, stories unmapped to units, unknown references, component dependency cycles — optionally scoped to one unit lane (its stories + contracts). All reads return CURRENT rows only (superseded rows from rewind/re-derive are filtered) and are sampled into the read ledger (DDB `READ#` rows).

5. **Prompt context.** The context compiler injects a bounded "Compiled graph context" pack into fresh stage prompts: input artifacts with enrichment gists, the unit lane pack (mapped stories, the requirements those stories cover via one `COVERS` hop, and touched contracts), decisions as cross-cutting constraints, a typed-item index with per-item traceability suffixes (`covers`/`persona`/`depends on`), and input-artifact TOCs — most load-bearing first, so the byte budget truncates navigation before substance.

6. **Verification.** Graph sensors gate stages: `required-sections` (headings, the `units:` DAG, structured-block shape — malformed blocks fail; ABSENT blocks are reported findings, flipped to blocking via the sensor row's `strictStructuredBlocks` switch once field-test compliance justifies it), `upstream-coverage` (citations), and `graph-coverage` (the intent-wide topology-integrity report over `get_coverage`, advisory by default).

7. **Audit.** `GET .../intents/{id}/audit` (the intent Audit page) aggregates the evidence: reads by tool, compact-read adoption share, enrichment spend, derivation health, structure-contract compliance rate, per-unit lane splits, sensor findings, and derived advisories (`context-heavy`, `enrichment-unused`, `structured-block-missing`, `derivation-failed`).

## Derive Enrichment Toggle

Derive-time enrichment is a **platform Admin setting** (Platform Admin → Agents), stored in the SSM parameter `/{project}/{env}/derive-enrichment` and editable at runtime — no redeploy. The value is snapshotted onto each execution's META row at intent create and forwarded by the orchestrator in the `derive-artifacts` payload, so a toggle flip applies to the next intent and never changes a run mid-flight.

- `llm` (deploy default): after a stage succeeds, the derive step makes one bounded, one-shot call per **changed** artifact (unchanged content keeps its summary — no repeat spend) through the already-configured agent CLI (same CLI/model selection as the run's stages — no separate inference path) to add summary metadata: a one-line `summary_gist` and key `summary_claims`. These props ride every compact read and the compiled context pack, and are part of the search corpus. Enrichment failures are logged and skipped; derivation itself never fails because of enrichment.
- `off`: derivation is fully deterministic. The graph has sections, typed items, citations, content hashes, and compact metadata. No extra model calls are made. (This is also the code-level fallback whenever the parameter is missing or unreadable — an ambiguous config read never spends tokens.)

The toggle never changes graph topology. Nodes, edges, citations, sensors, and typed item extraction are deterministic in both modes. Enrichment token/credit spend is recorded as metrics (priced at read time like stage spend, split out via the `enrichmentCalls` marker) and surfaced in the intent Audit view so its usefulness can be judged against the compact-read adoption it enables.

## Backfill

`POST /projects/{projectId}/intents/{intentId}/derive` (platform admin, refused while RUNNING) re-runs the projection over ALL of an intent's current artifacts — for intents predating the derivation feature, or to apply enrichment after flipping the toggle. Idempotent and safe to repeat.

## UI

The intent knowledge-graph page renders the derived layer (typed items + the unit-of-work DAG mirror, nodes tagged `graphLayer: 'derived'`) behind an "Artifacts / + Items & Units" toggle, including the item↔item traceability edges (`COVERS`, `FOR_PERSONA`, `DEPENDS_ON`, `IMPLEMENTS`, `EXPOSES`, `CONSUMES_CONTRACT`); `Section` nodes are deliberately not rendered. The Audit page is the measurement surface for the whole mechanism.
